redis主从复制

Posted on 2022-01-22
Words count in article: 25k | Reading time ≈ 25 mins.

主从复制

通过info命令查看redis状态

设从不设主

slaveof ip port

在某个redis实例中执行这个命令，就可以让这个实例成为一个redis实例的从机

然后主机保存的东西自动同步到从机

全量复制

一旦主从关系确定，首先会复制主机所有数据

增量复制

之后主库每次增加数据，都会同步到从库

读写分离

如果在从库执行写命令，会报错。从机不能进行写操作，只能进行读操作。而主机两者都可以，但master一般只进行写。

主机宕机

如果集群主机挂了，从机的info信息会显示主机挂了，集群不能进行写操作了，但还是可以从从机读取数据，从机原地待命

如果主机这时候恢复了，从机状态就会显示主机状态又为上线

从机宕机

如果从机宕机，主机状态就会显示从机少了一个

如果还有其他从机，其他从机不变

如果从机这个时候恢复，那么他会变成主机，不过下面没有从机【必须重新设置主从关系，执行从属命令】

如果挂机到恢复这段时间内，主机添加了新的数据 ，重新启动的该机器也会重新读取到，因为他是，重新把主服务器的数据读了一遍。

主机单机 从机上位

如果主机宕机恢复不了，想要有一个从机变成主机，其他从机隶属于它。

首先要断开和原来主机的主从关系，在某个从机上执行 slaveof no on，然后让其他的从机执行新的从属关系

slaveof ip port

如果从机很多，就得一个个来设置。

主机恢复，但是从机已经从属别人了

主机这时候已经变成空头司令，这个时候一般将他变成从机，加入新的主从关系

一台从机只能从一台主机，从机也可以有从机，构成树状结构

分担压力的同时，也增加了集群之间通信的时间

哨兵模式

redis哨兵模式就是启动一个进程监控master，如果master 挂了，在从机选举出一个主机，重新建立主从关系

可以参考默认的哨兵配置文件：sentinel.conf，这里我们自己创建一个

例如：我创建redis_sentinel.conf

编辑sentinel monitor dc-redis 127.0.0.1 6379 1

sentinel 表示哨兵配置

monitor 表示监控

dc-redis 哨兵名字，任意

127.0.0.1 ip

6379 端口

1 表示得到哨兵的投票数为1（当哨兵投票数大于或者等于该数字时候切换主从关系，此处哪一个从机最新达到1即可成为主机）

启动哨兵程序

redis-sentinel 配置文件目录

该程序也占用一个端口

也可以配置多个哨兵，每个哨兵监控一台机器

如果之前宕机的主机恢复了

采用了哨兵模式后，会让恢复的机器也可以变成一台从机，而不是像之前游离于整个程序之外

配置哨兵模式的规则

首先，主机死亡选举从机作为master，这个地方是由配置规则的。

优先级分别是

  1. 比较从机优先级。优先级的权重再配置文件中修改。redis.conf 的slave-priority（再redis高版本叫做replica-priority）

    这个地方设置的权重越小，优先级越高。默认100

  2. 比较偏移量。就是哪台redis的数据更加齐全。

  3. 比较runid。每台redis启动都会生成一串随机40位的runid。通过比较runid，来选举，其实就是随机看脸一样。

上面说命令slave-priority，在高版本变成replica-priority的原因是因为 slave这个词语过于敏感

slave lifes matter – 黑人的命也是名

所以几乎所有涉及到slave的命令和配置，不单单是redis，都尽可能地被改掉了。

redis集群

注意集群不是主从复制

redis集群传统集群：

客户端——>请求代理服务器（ngxin）——>转发到不同的redis

比如redis集群有3台，分别负责用户，订单，商品。一个用户的请求通过nginx转发的用户的redis。然后对应ngxin和redis可能会有集群和主从来确保高可用。

redis3.0推出了去中心化配置，可以不再引入nginx代理服务器。依然是上面的例子，一个用户的请求请求到3台redis中，如果这台redis不是负责用户的，它会将请求再转发到其他redis，即是每台都是中心。

例子：有3台主服务redis，分别对应用户，订单，商品。每台有一个从机，也是3台，一共6台。这6台就构成了一个redis集群。

集群相关的配置（在redis配置文件中，默认注释）

集群启动后，配置文件指定的cluster-config-file文件，就会生成。

最后将上述启动的redis示例合成一个集群，这里需要一个redis命令。

（这里是redis6，好像低版本还要安装rb安装环境）

构建好集群后，使用redis-cli 可以加上个 -c 参数（cluster缩写）采用redis集群的连接策略

插槽slots

集群中生成后，会有一件提示语 ：[OK] All 16384 slots covered.

16384是集群的插槽数目，一个redis集群固定有这么多插槽数目。在每个redis连接中，会根据计算插槽，将key保存到本台主机或是别的主机的插槽之中。

查看集群信息

cluster nodes 使用此命令可以查看集群信息

查看键所在槽位

cluster keysolt :key:

计算这个槽位中有多少个key

cluster countkeysinslot 槽位置号

获取指定插槽号中的指定个数的key

cluster getkeysinslot 插槽号 count

这里槽数量，查看时只能看自己当前连接的主机所分配的槽位，即便该槽位有:key:,但是时其他主机上，也会显示数量是0

redis集群异常处理

redis集群应该是已经集成了redis主从复制和哨兵

如果一个配置了从节点的redis主节点，宕机了，从节点就会在等待时间过后变为主节点。如果这个时候主节点又恢复了，主节点又会反而变成现在新的主节点的子节点。

如果一个集群中的一组主子节点都挂了，分两种情况

如果配置文件中该配置为yes，一组主从挂了，全部不能操作

如果为no，一组主从挂了，挂的部分不能操作
